## Project overview

Bootstrap compiler for **Syntact**, an experimental language where everything is a scope and execution is structural reduction. The compiler is written in Odin and targets x86-64 Linux. Syntact has no functions, classes, modules, or types as primitives — only scopes manipulated via binding, carving, extension, collapse, patterns, constraints, and (planned) effects/resonance/reactivity. See `README.md` for the full language design and `constraints.md` for the constraint/range system the analyzer implements.

Syntact's central model (from `README.md`):
- A **scope** `{}` is complete structured data; a file is the root scope.
- A function call decomposes into two independent operations: **carving** `scope{name -> v}` (derive a new scope) and **collapse** `scope!` (reduce through the scope's production).
- `:` is not a type annotation — it is **structural coloring**: a constraint propagates implicitly through the binding and everything it contains. Syntact has no type system.

## Build & run commands

```bash
# Build the compiler (the only consistently-building target)
odin build compiler -out:compiler/compiler

# Run the compiler on a .syn file
./compiler/compiler input.syn
./compiler/compiler input.syn --parse-only    # parse only, no analysis
./compiler/compiler input.syn --analyze-only  # parse + analyze, no reduction
./compiler/compiler input.syn --ast           # print AST
./compiler/compiler input.syn --ir            # print analyzer output (Type tree of root scope)
./compiler/compiler input.syn --print-errors  # show parse/analysis errors
./compiler/compiler input.syn --bc            # lower the reduced result to bytecode and print it
./compiler/compiler input.syn --regalloc      # bytecode annotated with the linear-scan allocation
./compiler/compiler input.syn --run 7 3       # interpret the bytecode; trailing args feed ??0, ??1, …
./compiler/compiler input.syn --emit -o prog  # emit an x86-64 ELF executable, then: ./prog 7
./compiler/compiler input.syn -t              # timing info (parse/analyze/reduce)
./compiler/compiler input.syn -v              # verbose pipeline logging

# Run the analyzer test suite (works from the repo root now — paths resolve to the source dir)
odin test test/analyze

# Run a single analyzer test by name (name is test_<stem>_<index> — find it in generated_tests.odin)
odin test test/analyze -test-name test_constraint_builtin_array_valid_0
```

## Architecture

Pipeline: **source → parser → analyzer (+ constraint folding) → reduce → bytecode → backend**. The bytecode + backends are the new live codegen path (the old `generate.odin` is dead, written for a deleted API). Two backends run today: a reference **interpreter** (`--run`, the oracle every other backend is validated against) and an **x86-64 ELF emitter** (`--emit -o prog`, producing a runnable static executable). Strings/floats are lowered for the interpreter; the x64 emitter currently handles the integer/bool domain (exit-status output).

The codegen bridge — what turns the reducer's output into machine code:
- **bytecode.odin** (~720 lines) — The **target-neutral bytecode**, the bridge between `reduce()`'s `^Type` DAG and every backend. SSA-like virtual registers (`BC_Value` = vN, defined once), so the reducer's CSE survives for free (lowering memoizes by DAG node address — a shared node → one vN). `lower_to_bytecode(reduce(scope))` is the one hard maillon, written once. Instructions: `BC_Const`/`BC_Const_F`/`BC_Str_Const` (int/float/concrete-string constants — integer constants are NEUTRAL, folded to immediates at the backend), `BC_Load_Arg` (a ??N → argv[slot], slot = `fixedpoint_id`), `BC_Bin`/`BC_Cmp` (arith/compare, domain on `value_types`), `BC_Move` (pattern phi merge), `BC_Label_Def`/`BC_Jump`/`BC_Branch_Zero`, `BC_Ret`. Each `BC_Value` carries a `Machine_Type` (enum: U8/I8/…/I64/F32/F64/Str) derived ONCE from `fold_type`/`cast_target` via `machine_type_of` — Syntact fixes width+domain semantically (structural coloring), so the bytecode PRESERVES it (a u8 stays U8, f32 vs f64 follow declared precision). A ??::u8 is normalized ONCE at entry (mask `& 0xff` unsigned, `shl/sar` sign-extend signed), then no further masking (the analyzer proved the downstream in range). A symbolic string (`"hi "+??::string`) or unsized domain is REJECTED with a clear error (`prog.error`), never lowered to a wrong `const 0`. `--bc` dumps it.
- **bc_interp.odin** (~330 lines) — The **reference interpreter**, the oracle: executes a `BC_Program` over tagged values (i64/f64/string), `--run a b …` feeding ??0, ??1, … as argv strings parsed per each `??`'s domain. Every machine backend must agree with it. A concrete string result is written to stdout; a float as a decimal; an integer plain. The `test/codegen` suite checks interp AND native against each case.
- **x64_regalloc.odin** (~250 lines) — Liveness (`[def, last_use]`, one pass) + **linear-scan** allocation over the SSA vN. 10 allocatable GPRs (RAX/RDX reserved as the imul/idiv scratch pair, RCX as shift-count scratch, RSP/RBP the frame, RBX reserved); spill to `[rbp-offset]`. `--regalloc` dumps the bytecode annotated with each vN's location. **Written and validated by dump but NOT yet wired into the emitter** — the live emitter uses the simpler stack-everything model (correct, passes all tests). The linear-scan is a pure perf optimization (no new case unlocked, real regression risk from fixed-register clobbers idiv/shift), so it is left ready-to-wire rather than risk the green suite.
- **x64_emit.odin** (~700 lines) — The **x86-64 emitter**: `BC_Program` → `.text` bytes + `.rodata` blob, via the tested assembler in `backends/x64` (encodings validated against objdump). Stack-everything model (every vN at `[rbp-8*(vN+1)]`, RAX/RCX/RDX work registers), two-pass label resolution (placeholder rel32 then patch). Covers: **integers** (exit-status output), **floats** (XMM `addsd/subsd/mulsd/divsd`, `cvtsi2sd`, decimal print = sign + integer part + 6 fraction digits), **strings** (`.rodata` pointer + `write(1,ptr,len)` syscall), **bool/none**. **Strength reduction + immediate folding** in `emit_bin`: `*2^k`→`shl`, `*3/5/9`→`lea`, unsigned `/2^k`→`shr`, unsigned `%2^k`→`and`, `+0/-0/*1` elided, `*0`→`xor`, constant operands → imm8/imm32 (the `emit_load_imm_rax` helper picks `mov eax,imm32` over `movabs` when it fits). Emits a runtime **arg stub** parsing `argv[1..]` (inline atoi) into a fixed `ARGS_TABLE`; `BC_Load_Arg` reads it by absolute address (a float ?? is `cvtsi2sd`'d — native float args are whole numbers, the interp handles full decimals). Adds `cqo`/variable-shift/atoi/print as raw bytes (the assembler lacks them).
- **x64_elf.odin** (~120 lines) — The **ELF64 writer**: `build_elf(code, rodata)` wraps the image as `[ELF header][program header][rodata][code]` — a minimal static ET_EXEC, one RWX PT_LOAD, entry at the code (right after rodata, so string literal addresses are known up front), `p_memsz` extended to cover `ARGS_TABLE` (at `ELF_BASE + 0x100000`, fixed so no chicken-and-egg with code length). `emit_executable(prog, path)` lowers → emits → writes the file `chmod 0755`.
- **x64_vectorize.odin** (~55 lines) — **Auto-vectorization analysis (no SIMD emitted yet, honestly).** Syntact's structural reduction folds all concrete computation at compile time, so the bytecode carries only SCALAR ops on individual `??` fixed points — no loop, no runtime iteration, no runtime-valued vector, hence nothing to vectorize (SIMD needs N independent elements under one op). `vectorizable_run` is the detector stub (always empty today); the file documents the contract for when the language grows runtime-iterable data (map/fold over many `??`, runtime lists, the planned effects/resonance) — then pack lanes into XMM/YMM and emit `paddd/addps` etc. (the assembler already has the encodings).

The compiler-internal `Machine_Type` / constants-as-immediates decisions are documented in the memory files; the key rule is that **constants carry no type — they fold into immediates at the backend** (a u8 calc stays u8, a list element of u8 is one byte; never promoted for "speed", which would break layout or wrap semantics).

### compiler/ package

All stages share one `package compiler`:

- **ast.odin** (~610 lines) — The AST **data model only**, split out of `parse.odin`: the `Node_Kind`/`Literal_Kind`/`Operator_Kind` tags, the `*_Data` payloads and the `Node_Data` raw union, the flat SOA `Ast` container (`node_kinds`/`node_data`/`node_spans`/`extra` indexed by `Node_Index`, `INVALID_NODE` sentinel), `Span`/`Position` + span→position mapping, and the `node_*` accessors + `print_ast`/`ast_root`. No lexer/parser logic. Read a node through the `node_*` accessors, never the raw union.
- **parse.odin** (~2830 lines, formerly `parser.odin`) — Lexer + parser **machinery only** (the AST it builds lives in `ast.odin`). Table-driven lexer (one handler per leading byte), single-pass top-down Pratt parser, covers the full grammar: bindings (`->`/`<-`), productions, carving (`{}`), extension (`+{}`), collapse (`!`), patterns (`?`), constraints (`:`), raw casts (`::`), ranges (`..`), events (`>-`/`-<`), resonance (`>>-`/`-<<`), reactivity (`>>=`/`=<<`), property access (`.`), externals (`@`), operators. Entry point `parse(cache, source)`. `::` lexes as a single `Cast` token (intercepted in `lex_colon` before the `:` whitespace table) and parses as an infix binary `Operator_Kind.Cast` at `CONSTRAINT` precedence — tighter than `+`, so `a+b::u8` is `a+(b::u8)` and casting a sum needs parens: `(a+b)::u8`.
- **ir.odin** (~720 lines) — The IR **data model + its printing**, split out of `analyze.odin` (data) and `type.odin` (printing), mirroring how `ast.odin` carries both the AST structs and `print_ast`. Holds the `Type` union and every variant (`Integer_Type`, `Float_Type`, `String_Type`, `Bool_Type`, `Scope_Type`, `Range_Type`, `Or_Type`/`And_Type`/`Negate_Type` for `|`/`&`/`~`, `Compose_Type` for arithmetic, `Cast_Type` for the `::` raw cast, `Pattern_Type`/`Pattern_Branch` for `target ? {…}` (folded/checked in pattern.odin), `Carve_Type`, `Mention_Type`, `Reference_Type`, `None_Type`, `Unknown_Type`, `Invalid_Type`), the domain interval payloads (`Integer_Interval`/`Float_Interval`/`String_Interval`, `FloatKind`), `Binding_Kind`, plus the renderers: `print_type` (the `--ir` tree dump), `value_to_string`/`write_value` (concrete reduced/default value, used by the reducer tests — `write_value`'s `Carve_Type` case folds the carve via `fold_carve` and renders the substituted scope, so a carve's default is the default of its resulting scope), `type_to_string` (folded type for error messages), and `op_symbol`/`print_*_interval`. Per-domain rendering still lives in each domain file (`integer_to_string`, `bool_to_string`, …); this file composes them.
- **analyze.odin** (~1025 lines, formerly `analyzer.odin`) — Semantic analysis **logic** (the IR types it builds live in `ir.odin`). Builds a tree of `Scope_Type` (parallel `[dynamic]` arrays: `names`, `types` (constraints), `kind` (`Binding_Kind`), `values`, plus `type_folds`/`constraint_folds`). `walk()` is the AST→`Type` recursion, but is now a thin **dispatcher**: it switches on `Node_Kind` and routes each case to its own `walk_<kind>` handler (`walk_scope_node`, `walk_binding`, `walk_carve`, `walk_property`, `walk_operator`, `walk_literal`, …), one per branch — same split style as the Pratt parser's prefix/infix handlers. The handlers carry the per-kind logic; add a node kind by adding a `case` + a `walk_<kind>` proc. (`walk_operator` routes `&`/`|`/`~` to the set-algebra variants, `::` to a `Cast_Type` folded by `fold_cast`, and everything else to a `Compose_Type` folded by `fold_compose`.) `analyze()` is the entry; `follow()` chases references; `sem_error()`/`sem_warning()` take the offending node's **`Span`** (via `node_span(a, idx)`) and record an `Analyzer_Error` carrying BOTH that `span` (the range to underline) and its resolved `position` (line/column/offset) — mirroring `Parse_Error`, so the LSP and any consumer read the range directly without recomputing. (the `Analyzer_Error`/`Analyzer_Error_Type` diagnostics stay here, with the analyzer state). The result lands in `cache.scope` / `cache.analyze_errors`. Two carve-related pieces live here: (1) `walk_property` resolves a **source-none property** (`.x`, `.#0`, `.x#0` — a self-mention written with no left side) against the scope being carved, reading its *original* field value, via `self_resolve` (a parent-less local lookup) and the `carved_scope` field on `Analyzer` (set/restored around each override walk in `walk_carve`, so nested carves resolve `.` against the nearest one); (2) `recheck_carve` folds a carve via `fold_carve` and re-proves every colored binding against its substituted value, raising `Constraint_Mismatch` for **implicit constraints** — a constraint on one field broken because another field it references was carved out of range (`u8:y -> x` then `a{x -> -11}`). The directly-overridden fields are still checked inline in `walk_carve`; `recheck_carve` covers the dependent ones.
- **integer.odin** (~800 lines, formerly `fold_integer.odin`) — Interval arithmetic over `Integer_Interval{lo, hi: Maybe(i64)}` (nil = ±∞), implementing `constraints.md`. Folds compose/range types to `[]Integer_Interval`, does interval add/sub/mul, union/intersect/negate/normalize, and `integer_intervals_satisfy` for the subset check (the constraint-as-contract proof). `fold_type_intervals` has a `Cast_Type` case so an INLINE `::` (`(??::u8)`) folds to the target's envelope (u8 → 0..255, or the cached concrete value when the source was concrete) — the same envelope a *reference* to a `::`-bound binding yields. This is what makes `(??::u8)*2` fold to `0..510` exactly like `a -> ??::u8 ; a*2` (without it the inline cast returned nil and `*` raised a spurious `Invalid_operator`). `float.odin`'s `fold_type_float_intervals` mirrors this with its own `Cast_Type` case. `builtin_name`/`builtin_alias` map intervals back to `u8`, `i32`, etc.; `int_layout` maps an interval to its `Int_Layout{bits, signed}` (the fixed-width builtins only). Also home to the **raw-cast bit layer** shared by `::` (see type.odin): `Bit_Repr{bits, width, signed, from_float}` is the little-endian pattern of any concrete value; `resize_bits` pads/cuts it (zero/sign-extend, truncate) to a target width; `bits_reinterpret_int` reads a resized pattern back as a two's-complement integer.
- **float.odin** (~640 lines) — The float domain, mirror of `integer.odin` over `[]Float_Interval`. Interval arithmetic and subset checks for the float family. `fold_type_float_intervals` has a `Cast_Type` case (envelope propagation, like integer.odin). Interval `Multiply` is computed over the EXTENDED reals (`ext_mul`, with `0·∞ = 0`, and a missing bound = ±∞), so an UNBOUNDED fixed point `??::f64` = (-∞,+∞) multiplies by a constant and stays a (still unbounded) interval instead of failing the fold — without this `x*2.0` (x a float `??`) raised a spurious `Invalid_operator` while `x+1.0` worked (Add already handled open bounds). `Divide` still bails on an open bound.
- **string.odin** (~1170 lines) — The string/char family over `[]String_Interval`. **Mode** is the interval's own `ordinal: bool` (set once at construction by `quotation_is_ordinal` from the literal's quote+length — single-quote ≤ 1 codepoint → ordinal `'a'..'z'`, everything else → positional `"a".."z"` prefix/suffix); `string_interval_mode` just reads the flag. There is no quotation stored in the interval anymore, and `String_Type` has **no `is_sequence` flag**. A `[]String_Interval` is **always a union** of alternatives (`|`). The ordered concatenation `+` is NOT flattened into a union: it stays a `Compose_Type` and is matched in order at satisfy time. `fold_string_sequence` collects the `+` chain's segments left-to-right (`"id_" + '0'..'9'*1..` → `[{"id_"}, {digits *1..}]`); `string_compose_satisfy`/`string_value_satisfies_sequence`/`seq_match` walk a concrete value, matching each segment its `count` times with backtracking. A **purely positional** `+` (every segment a concrete/prefix/suffix, no ordinal class, no repetition) still flattens to the legacy `{prefix, suffix}` interval via `positional_concat`/`concat_positional_fold`, so `~("".. + '_')` keeps working and `'a'..'z'+'@'+'a'..'z'` has a materializable default (`string_sequence_default`). **Negation in a sequence** works: an ordinal negation (`~'a'..'z'`) folds to codepoint-complement intervals (`string_intervals_negate`, consumable as a normal segment); a positional word negation (`~"piro"`, `~"x"`) becomes a SEGMENT-NEGATION sentinel — a `String_Interval` whose `count` is the impossible `{-1..-1}` (tagged by `seg_is_negation`/`count_negation_sentinel`) and whose `lo` is the forbidden word — which `seq_match` matches as a variable-length prefix that is NOT that word. The **three-bound range** `"ab".."cd".."ef"` stays a raw `Range_Type` (`string_is_tri_range`); `string_tri_range_satisfy` enforces all three: starts with the prefix, CONTAINS the middle, ends with the suffix. The compose/tri-range constraints are kept un-flattened by `fold_constraint` returning the raw node, and `satisfy` (type.odin) routes a `Compose_Type`/`Range_Type` `fc` to the sequence/tri matchers. `range_operand_kind` recognizes `Negate_Type`/`And_Type`/`Or_Type` bounds so `(~'\0')..` reads as a string range.
- **bool.odin** — The boolean domain, a finite set over `{true, false}`. `Bool_Type{value: Maybe(bool), default: bool}`: `value` is the concrete element for a singleton, `nil` for the full `{true,false}`; `default` is the materialized default — the **first source term** of the domain (`true` → true, `false` → false, `(true|false)` → true, `(false|true)` → false, the `bool` builtin → false). `&`/`|`/`~` are **set operations** (intersection/union/complement within `{true,false}`), not runtime logic: `true & false` → `none` (empty set), `(true|false)` → `{true,false}`, `~true` → `{false}`. The empty set folds to `None_Type`. Mirrors the string-family entry points (`fold_type_bool`/`fold_constraint_bool`/`bool_satisfy`).
- **type.odin** (~770 lines, formerly `fold_type.odin`) — Generic, domain-agnostic fold dispatch **only** (the `Type` rendering moved to `ir.odin`): `fold_type` derives the constraint a value *produces* (its envelope), routing to the integer/float/string/bool domains; `fold_constraint`/`fold_value_type`/`satisfy`/`type_default`/`default_value` resolve and check constraints. A collapse (`target!`, an `Execute_Type`) folds on both sides to the corresponding fold of the FIRST production it reduces through — `execute_production` resolves the target to a scope (peeling a carve) and returns its first `.Product` value, mirroring `reduce`/`execute`; `fold_constraint` and `fold_value_type` then recurse on it. Higher-level `fold_compose`/`fold_range` (validates range operand kinds and color matching — integer/float/string, no cross-family) plus type-shape checks. Emits `Invalid_operator`/`Invalid_Range` errors when operands can't fold — EXCEPT a **comparison** (`<`,`>`,`<=`,`>=`,`==`,`!=`, via `is_comparison_op`) over compatible numeric families is NOT an error even though it never folds to a numeric envelope (it yields a bool): `fold_compose` leaves it un-folded (no `type_fold`) and the reducer handles it — `n < 5` (n a `??`) reduces to the symbolic `??0 < 5`, a concrete pair to a bool. `diagnose_compose`/`family_of` see a `Cast_Type` as its target's family (so `??::u8` is Integer, not an opaque "value", letting a fixed point participate in arithmetic/comparison). `fold_cast` resolves a `value :: target` **raw binary reinterpret-cast** (domain-agnostic): the target must fold to a single fixed-width layout (u8/i32/f64/bool/char or a string) via `cast_target`, else `Invalid_Cast`. `cast_to_bits` extracts the concrete source's little-endian bits (reading its width from its value, or from its color via `source_color`/`colored_int_layout`/`colored_float_kind` for a bare literal under a sized constraint); `cast_from_bits` resizes (pad/cut) and lays the pattern back down in the target domain. The result's envelope is always the target, so `fold_constraint`/`fold_value_type` return the target for a `Cast_Type` (the cast forces the value in — it never raises `Constraint_Mismatch`). Bit-for-bit for int/bool/string and float↔non-float; float→float of a different width is a **value** conversion (IEEE rounding, `f64 1.0 :: f32` → 1.0f) via `Bit_Repr.from_float`. So `(a+b)::u8` makes a u8-overflowing sum satisfy `u8:c`, and `??::u8` is an unknown whose envelope is `u8`. `fold_carve` materializes a `Carve_Type` into its substituted `Scope_Type` (clone the source scope, write each override into the field it targets + refresh that field's cached `type_fold`, then `repoint` every `Mention`/`Reference` that named the source scope so it names the copy — sibling references then read the substituted values, cascading transitively); it is pure (no diagnostics) and is wired into both `fold_type`/`fold_value_type` and `fold_constraint`. `repoint` is a copy-on-write rewrite of a `^Type` tree (descends through composites and nested scopes, clones a node only when a descendant changed). `scope_clone` copies a scope's columns; `new_type` boxes a `Type`.
- **pattern.odin** (~200 lines) — The pattern domain — `target ? { match -> product, … }` (the `Pattern_Type`/`Pattern_Branch` IR lives in ir.odin). A branch has two modes: **typecheck** `M -> p` (the match `M` is a constraint; fires when `type_fold(target) ⊆ M`) and **value** `=v -> p` (the match is a value; fires when `type_fold(target) ⊆ {-> v}` — `walk_pattern` peels the unary `=` operator and sets `value_match`). A `nil` match (`-> p`) is the default and matches anything. **Exhaustiveness** (`pattern_is_exhaustive`): a default branch, OR the OR of all branch matches typechecks against the target — `satisfy_root(fold_constraint(Or of covers), fold_value_type(target))`, where a value branch enters the union bundled as `{-> v}` (via `make_producer_scope`, `branch_match_cover`) and a typecheck branch enters as `M`. So `(0..120 | 22..255)` covers a u8 target just as the binding `(0..120|22..255):data -> ??::u8` typechecks; an uncovered target raises `Non_Exhaustive_Pattern`. **Folds**: `fold_constraint_pattern` → the product of the FIRST branch that covers the target (deterministic); `fold_type_pattern` → when the target is a CONCRETE singleton (`pattern_target_is_concrete`) the first branch that matches in order wins (`5 ? {=7 -> 100, -> 0}` → 0); when the target is a SET (a range / `??`) it is deterministic ONLY if the **first** branch covers the whole target (a covering branch — e.g. a default — sitting AFTER partial branches is NOT deterministic, since earlier branches intercept some values), else the **combined** `Or` of every branch's product. (A subtle bug fixed: returning the first *covering* branch made `?? : u8 ? {0..127 -> 0, -> 1}` fold to `1`, dropping the `0` branch — it must be `0 | 1`.) `branch_covers` is the per-branch firing test (`satisfy_root` of the branch's match against the target's type_fold). A Pattern_Type NEVER survives in a fold — it always resolves to a branch product (or its combined Or). `reduce_pattern` (reduce.odin) collapses to the firing branch's product; an unreachable fallback returns `Invalid_Type`, never the pattern.
- **diagnostics.odin** (~351 lines) — Turns analysis failures into author-facing messages. The fold procedures answer a yes/no question (does this resolve?); this layer explains *why* it didn't.
- **reduce.odin** (~780 lines) — **Symbolic fixed-point reduction (DAG + constant folding + CSE).** `reduce(scope) -> ^Type` collapses a scope through its `.Product` binding, but reduction is PARTIAL EVALUATION over the program's fixed points — each `??` (an external the linker resolves: argc, an env value, …) is a free variable the reducer carries rather than evaluates. THE RULE, applied to every node: (1) look at its `fold_type`; (2) if that fold is a **concrete singleton** it already IS the reduction — return it, no evaluation (`singleton_shortcut`); (3) otherwise a fixed point survives, so recurse into the node's **value** and partially evaluate, isolating the `??` so the runtime does the minimum work. `reduce_value` is the recursive evaluator (Execute/Compose/Cast/Carve/Mention/Reference/Scope/Or/And/Negate/Pattern + the domain leaves); `is_fixed_point` decides whether a node bottoms out in an Unknown. **We want to SEE the `??`, not the aliases**: `reduce_mention`/`reduce_reference` ALWAYS follow a name to its bound value and reduce through, and when the value bottoms out in a fixed point they descend to the ATOMIC `??` node (`follow_to_fixedpoint`) — so every route to one unknown converges on one node, hence one `??N`, and `c -> 2*n` + `e -> n` collect (`c + e` → `3 * ??0`). The only symbolic leaf that survives is the atomic terminal `??` (a `Cast(Unknown)`/`Unknown`). **Crucially the reducer does NOT distribute** — distributing a product of sums multiplies the term and multiplication count, the opposite of operation-minimal. `reduce_compose` keeps the author's FACTORED structure and only: FOLDS CONSTANTS (`eval_concrete` when all operands concrete; `simplify_arith`/`fold_const_into_mul`/`scale_node`/`offset_node` coalesce constants along a `*` chain — `5*e*14`→`70*e`, `a*2*2`→`4*??`), COLLECTS ADDITIVE LIKE TERMS along a `+`/`-` chain (`collect_sum`/`flatten_sum`/`rebuild_sum`: `k*x + m*x`→`(k+m)*x`, the constant folded — `2*n + n`→`3*??0`, `(sub+3)-4`→`sub-1` — this REDUCES ops and is NOT distribution; products of sums stay factored), ELIMINATES IDENTITIES (`*1`,`+0`,`x-x`→0,`*0`→0), and SHARES COMMON SUBEXPRESSIONS via `dag_intern` (hash-cons by structural `dag_key`, so `e` and `(…)*e` are built once — the CSE codegen wants). A non-ring operator (`/`,`%`,comparison,bit op) stays a symbolic node with reduced operands. So `((a*2)+3-4)*3` (a→`??::u8`) → `3 * (2 * ??0 - 1)` (factored, not the distributed `6*??0-3`); `(b+2+5*e)*14*e` with `b→a*2, a→??::u8*2, e→??::u8` → `14 * (4 * ??0 + 2 + 5 * ??1) * ??1`. Fixed points render as **`??N`** (`fixedpoint_id`: a stable, appearance-ordered index keyed by the `??` node address or a name's `(scope,index)` site) — so two mentions of one unknown collect (`a+a`→`2 * ??0`) while two DISTINCT `??` stay apart (`a+b`→`??0 + ??1`, the linker resolves each separately). `dag_key` is the structural identity (commutative ops sort their operands) used for both CSE and the `x-x`/`x+x` tests. `reduce_pattern` resolves the firing branch when the target is concrete, else reduces each branch product and keeps the pattern shape; `reduce_set_op`/`execute`/`reduce_carve` handle `|`/`&`/`~`, collapse, and carve. The concrete evaluators (`compose_arith`/`compose_eq`/`compose_ord`/`compose_bitlogic`/`compose_shift`, via `eval_concrete`) run on already-reduced concrete operands. The result renders through `value_to_string`/`write_value` (ir.odin) — the reduce CLI output and the tests BOTH use this path (resolve.odin uses `value_to_string`, not `print_type`, for the reduced value); `write_compose_value`/`write_operand_value` parenthesize by operator precedence (`op_prec`) so `(??0 + 1) * ??1` is not mis-read, and the fixed-point cases render `??N`. **GOTCHAS**: the DAG bookkeeping (`dag_table`/`fixedpoint_index`/`fixedpoint_next`) is `@(thread_local)` (the test runner reduces on multiple threads — a shared global map data-races into a segfault) and reset each `reduce()` (`dag_reset`); `dag_key` strings are heap-allocated (`fmt.aprintf`, not `tprintf`).
- **resolve.odin** (~840 lines, formerly `resolver.odin`) — Compilation orchestrator. Thread pool for parallel files, `Cache` struct per file (`scope`, parse/analyze errors+warnings, `status`). Entry point `resolve_entry()`; the per-file pipeline that wires parse → analyze → (`--ir` print) → reduce → `print_type` is around line 370.
- **main.odin** — CLI; `parse_args()` into the `Options` struct. On-disk caching is force-disabled in the bootstrap (`no_cache = true`).
- **generate.odin**, **backends/x64|wasm|arm64/** — x64/codegen, WIP and not in the active pipeline.

Key conventions:
- **Builtin names are lowercase.** The generic-domain builtins are `int`, `float`, `string`, `bool`, `none` (registered in `init_builtins`), alongside the fixed-width `u8`/`i32`/`f64`/… The open-domain printers (`integer.odin`/`float.odin`/`string.odin`/`bool.odin`) print these same lowercase names, so source, IR, and diagnostics agree. There is no `Int`/`Bool`/`String` — a capitalized name like `Array` or `F32ORString` is always a user-defined scope, never a builtin.
- The constraint system is the heart of the analyzer. A concrete value `5` is the degenerate range `5..5`; `u8` is `0..255`; `true` is the singleton `{true}`. Constraints are **static contracts** — the analyzer must *prove* `value ⊆ constraint` (via `integer_intervals_satisfy`, `bool_satisfy`, …) or it's a `Constraint_Mismatch`. There is no implicit coercion; `&` is the explicit **narrowing** cast/intersection (it must still prove `value ⊆ result`). Distinct from it, `::` is the **raw binary reinterpret-cast**: `value :: target` forces the value's bits into the target's layout (pad/cut + reinterpret), so it always lands in the target and never needs a proof — `&` narrows, `::` reinterprets. Read `constraints.md` before changing folding.
- Same-name bindings are valid and tracked by ordinal (`#0`, `#1`, …) — a core feature, not a bug. Access (`.`) resolves the **last** occurrence; carving (`{}`) targets the **first** by default.
- Errors use the `Analyzer_Error_Type` enum and the term **constraint** (`Constraint_Mismatch`, `sem_error`-style naming), never "type"/"typecheck" — Syntact has no types.
- A constraint must denote a **statically-known set**. `Insoluble_Constraint` is raised when a constraint depends on an unknown value (`??`) — directly, through a reference, or inside any composition (`&`/`|`/`~`/arithmetic/range/carve/execute/pattern). The distinction is sharp: `u8` is the *set* `0..255` (solvable), but `??::u8` is *one indeterminate element* of u8, so `??::u8:a -> 10` cannot be proven. `constraint_depends_on_unknown` (analyze.odin) walks the raw constraint tree (following references, with a depth guard) and treats a `??` as solvable only when a `::` pins it to a concrete singleton. `Insoluble_Constraint` is checked in `typecheck` **before the `fc == nil` bail and the satisfaction proof**, so an unknown buried under any operator is caught even when the constraint folds to nil (e.g. a pattern whose target is a bare `??`) — but a `??` on the **value** side (`u8:a -> ??::u8`) is fine, only the constraint side must be solvable.
- **Or on the value side.** `satisfy(fc, ft)` decomposes a union on the VALUE side conjunctively: `a | b` satisfies `fc` iff BOTH `a` and `b` do (every value the union can take must land in `fc`). This mirrors the disjunctive Or on the constraint side and is what lets a pattern's combined product type `"" | 10` prove against the color `"" | 10`. Checked first, before the switch on `fc`.
- **Producer self-match vs raw productions.** `satisfy_root` folds each production it compares (`fold_constraint(v.values[i])`) before matching, because a producer scope built from a literal (`{->0..}`) stores its production RAW (a `Range_Type`), which `satisfy` cannot match against an `Integer`. Folding normalizes it to its domain set. This does NOT weaken the self-match rule: `u8:a -> u8` / `0..255:a -> 0..255` still fail (a set is not an element of itself), but `{->0..}:a -> 0..` (a producer of a set, satisfied by that set) now passes — needed for value-match exhaustiveness `0.. ? { =0.. }`.
- **Carve property values.** `reference_effective_value` (type.odin) resolves a `Reference_Type` honoring a carve in its target: a property `C.x` where `C` is `Base{x -> 10}` reads the OVERRIDE (10), not the original `Base.x` (5). `fold_value_type`'s Reference case uses it, so a carved field's value folds to the substituted value (mirrors `reduce_value`). `fold_constraint`'s Reference case keeps reading the original COLOR — a carve overrides the value, not the constraint.
- **Un-flattened string constraints.** A string `+` sequence and a three-bound string range do NOT fold to a flat `String_Type` — `fold_constraint` returns the raw `Compose_Type`/`Range_Type`, and `satisfy` has a `Compose_Type` case (→ `string_compose_satisfy`, ordered sequence) and a `Range_Type` case (→ `string_tri_range_satisfy`, prefix/contains/suffix). `fold_compose` does not diagnose a non-collapsing string `+` (it is a valid sequence). See string.odin.

### test/ package — six independent harnesses

`test/parse/`, `test/analyze/`, `test/typecheck/`, `test/reduce/`, `test/default/`, `test/codegen/` are separate Odin test packages (packages `parse_test`/`analyze_test`/`typecheck_test`/`reduce_test`/`default_test`/`codegen_test`), each with `tests/*.json`, a `generated_tests.odin`, a `generator.odin`, and a runner. Run each with `odin test test/<category>` from the repo root (or `cd test/<category> && odin test .`) — the runner's `test_path` helper resolves `tests/*.json` against its own source directory, so the CWD no longer matters. The `generator.odin` regeneration step, by contrast, still reads/writes relative to the CWD and must run from inside the category directory.

- **codegen** — the exhaustive **end-to-end** suite (44 cases, all pass). Each case has `source` + `args` + `expect` + `kind` (`int`/`str`/`float`/`reject`). The runner lowers to bytecode, then validates BOTH backends: the **interpreter** (`interp_bytecode(args)` vs expect — the oracle) AND the **native x64** (emit ELF to a temp file, run it via `popen`, compare exit-status for int/bool, stdout for str/float, numerically with tolerance for float). A `reject` case asserts lowering fails (symbolic string etc.). An interp/native divergence fails the case. Covers every domain and construct: integer arithmetic + all strength reductions (`*2^k`/`*3,5,9`/`/2^k`/`%2^k`), u8 wrap, patterns, multi-arg, comparisons, carve/collapse/refs/set-ops/bool, strings (literal/concat/pattern/ref), floats (add/mul/div/sub/symbolic). Uses libc `popen` to run the produced binaries.

- **parse** — JSON has `expect`, a serialized AST like `Scope[Operator(Add,Literal(Integer,1),Literal(Integer,2))]`. *Builds and runs* (1239 cases; ~10 fail on error-recovery / ambiguous-pattern cases).
- **analyze** — JSON has `expect_errors: []string` of `Analyzer_Error_Type` names (empty array = must analyze cleanly). The runner parses, analyzes, and compares the error list. *Builds and runs.*
- **typecheck** — same JSON shape as analyze (`expect_errors`), focused on constraint-satisfaction cases (`(string|u8):b -> "hi"`, `~10:a -> 5`, …), including carve **implicit-constraint** cases (`u8:y -> x` then `a{x -> -11}` → `Constraint_Mismatch`, transitive and `x+y` variants), **raw-cast** cases (`(a+b)::u8` satisfies `u8:c`; `??::u8`; `Invalid_Cast` for a layout-less target like `10..37`/`int`/`float`/disjoint union/negation), **composite** cases (mixed-family unions `(u8|f32|string)`/`(bool|u8)`, composed/double/triple negation `~(5|10)`/`~~~10`/`~(u8 & ~5)`/De Morgan `~(~(0..10) | 20)`, nested `&`/`|` of ranges and comparisons, **scope-shape** constraints `{u8:x}:a -> {x -> 5}` via `scope_satisfy`, computed scope fields `{u8:x -> 200, u8:y -> x+x}`, properties inside computations `pack.x + 50`), **self-match** cases (a non-singleton set never satisfies itself — `u8:a -> u8`, `string:a -> string`, `C -> u8 ; C:a -> C` all fail; a singleton does — `5:a -> 5`, `C -> 5 ; C:a -> C` pass), **carve** cases (carve as a constraint `Base{x->5}:a -> {x -> 5}`, via a reference `Narrowed -> Base{x->5}`, nested carves, reference-valued overrides `Base{x->k}`, dependent-field implicit checks, property of a carved scope), **execute** cases (`Base{x->5}!` as value and as constraint, collapse via reference chains `Q -> P ; Q!`, a producer of a SET vs a producer of a concrete element), and **references composed in `&`/`|`/`~` and arithmetic** (`(C & >10)`, `(C | string)`, `~C`, `A + B` under a constraint), plus the `tc_pat_*` **pattern** cases and the `tc_str_neg_*`/`tc_str_tri_range_*` **string sequence/negation/three-bound-range** cases. *Builds and runs — all 460 pass.* (The string gaps that `constraints.md` flagged are now implemented, not just exposed.)
- **reduce** — JSON has `expect`, the stringified reduced value. The runner calls `reduce(cache.scope)` then `value_to_string`. `expect` is a concrete value (`"25"`) OR a **symbolic reduced expression** (`"3 * (2 * ??0 - 1)"`) when the program has surviving fixed points. *Builds and runs*; some cases still fail on unimplemented reducer/parser features (see "Current state").
- **default** — JSON has `binding` (a name) + `expect` (the materialized default, rendered by `value_to_string`). The runner analyzes, finds the **last** binding of that name in the root scope, and compares its `type_folds` value (or `values` if unfolded) to `expect`. Covers the default-materialization rule (range → low bound, union `|` → first term's default, negation `~X` → first element outside X, scope → `{name -> default, …}`, carve → default of the substituted scope, reference-as-type → default of the referenced set, execute → default of the produced value) across all domains and composites, including carve-via-reference, nested carves, and collapse-via-reference. *Builds and runs — all 109 pass.* The materialization gaps it used to expose are now fixed (cross-family union defaults via `type_default`'s `Or_Type` first-term rule, computed scope/carve fields via `write_value` preferring `type_folds` and `fold_carve` re-folding dependents, empty-set negation → `none`, string repetition/quoting, and string negation → first string outside the set).

### lsp/ package

- **lsp.odin** (~1614 lines) — LSP server (JSON-RPC over stdin/stdout): `initialize`, document sync, **diagnostics** (parse + analyze errors/warnings straight from the `Cache`, each rendered to a precise LSP range via `span_range` from the error's `span` — trimmed of trailing whitespace so the highlight never bleeds onto the next line), **hover** (`handle_hover` — name + binding-kind symbol + folded constraint/type via `describe_type`), **go-to-definition**, **find-references**, **rename**, **completion** (scope bindings + builtins, with folded-type details), and **semantic tokens**. - **semantic.odin** (~318 lines) — the LSP's semantic layer over the NEW model.
