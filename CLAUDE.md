## Project overview

Bootstrap compiler for **Syntact**, an experimental language where everything is a scope and execution is structural reduction. The compiler is written in Odin and targets x86-64 Linux. Syntact has no functions, classes, modules, or types as primitives — only scopes manipulated via binding, carving, extension, collapse, patterns, constraints, and (planned) effects/resonance/reactivity. See `README.md` for the full language design and `constraints.md` for the constraint/range system the analyzer implements.

Syntact's central model (from `README.md`):
- A **scope** `{}` is complete structured data; a file is the root scope.
- A function call decomposes into two independent operations: **carving** `scope{name -> v}` (derive a new scope) and **collapse** `scope!` (reduce through the scope's production).
- `:` is not a type annotation — it is **structural coloring**: a constraint propagates implicitly through the binding and everything it contains. Syntact has no type system.

## Build & run commands

```bash
# Build the compiler (the only consistently-building target)
odin build compiler -out:compiler/compiler

# Run the compiler on a .syn file
./compiler/compiler input.syn
./compiler/compiler input.syn --parse-only    # parse only, no analysis
./compiler/compiler input.syn --analyze-only  # parse + analyze, no reduction
./compiler/compiler input.syn --ast           # print AST
./compiler/compiler input.syn --ir            # print analyzer output (Type tree of root scope)
./compiler/compiler input.syn --print-errors  # show parse/analysis errors
./compiler/compiler input.syn -t              # timing info (parse/analyze/reduce)
./compiler/compiler input.syn -v              # verbose pipeline logging

# Run the analyzer test suite (works from the repo root now — paths resolve to the source dir)
odin test test/analyze

# Run a single analyzer test by name (name is test_<stem>_<index> — find it in generated_tests.odin)
odin test test/analyze -test-name test_constraint_builtin_array_valid_0
```

### Regenerating test harnesses

Each test category has its own `generator.odin` (marked `+build ignore`) that scans `tests/*.json` and rewrites `generated_tests.odin` (auto-generated — never hand-edit). Run from inside the category directory:

```bash
cd test/analyze && odin run generator.odin -file && cd -
# same pattern for test/parse, test/typecheck and test/reduce
```

## Architecture

Pipeline: **source → parser → analyzer (+ constraint folding) → reduce**. Code generation (`generate.odin`, `backends/`) is WIP / inactive.

### compiler/ package

All stages share one `package compiler`:

- **ast.odin** (~610 lines) — The AST **data model only**, split out of `parse.odin`: the `Node_Kind`/`Literal_Kind`/`Operator_Kind` tags, the `*_Data` payloads and the `Node_Data` raw union, the flat SOA `Ast` container (`node_kinds`/`node_data`/`node_spans`/`extra` indexed by `Node_Index`, `INVALID_NODE` sentinel), `Span`/`Position` + span→position mapping, and the `node_*` accessors + `print_ast`/`ast_root`. No lexer/parser logic. Read a node through the `node_*` accessors, never the raw union.
- **parse.odin** (~2830 lines, formerly `parser.odin`) — Lexer + parser **machinery only** (the AST it builds lives in `ast.odin`). Table-driven lexer (one handler per leading byte), single-pass top-down Pratt parser, covers the full grammar: bindings (`->`/`<-`), productions, carving (`{}`), extension (`+{}`), collapse (`!`), patterns (`?`), constraints (`:`), raw casts (`::`), ranges (`..`), events (`>-`/`-<`), resonance (`>>-`/`-<<`), reactivity (`>>=`/`=<<`), property access (`.`), externals (`@`), operators. Entry point `parse(cache, source)`. `::` lexes as a single `Cast` token (intercepted in `lex_colon` before the `:` whitespace table) and parses as an infix binary `Operator_Kind.Cast` at `CONSTRAINT` precedence — tighter than `+`, so `a+b::u8` is `a+(b::u8)` and casting a sum needs parens: `(a+b)::u8`.
- **ir.odin** (~720 lines) — The IR **data model + its printing**, split out of `analyze.odin` (data) and `type.odin` (printing), mirroring how `ast.odin` carries both the AST structs and `print_ast`. Holds the `Type` union and every variant (`Integer_Type`, `Float_Type`, `String_Type`, `Bool_Type`, `Scope_Type`, `Range_Type`, `Or_Type`/`And_Type`/`Negate_Type` for `|`/`&`/`~`, `Compose_Type` for arithmetic, `Cast_Type` for the `::` raw cast, `Pattern_Type`/`Pattern_Branch` for `target ? {…}` (folded/checked in pattern.odin), `Carve_Type`, `Mention_Type`, `Reference_Type`, `None_Type`, `Unknown_Type`, `Invalid_Type`), the domain interval payloads (`Integer_Interval`/`Float_Interval`/`String_Interval`, `FloatKind`), `Binding_Kind`, plus the renderers: `print_type` (the `--ir` tree dump), `value_to_string`/`write_value` (concrete reduced/default value, used by the reducer tests — `write_value`'s `Carve_Type` case folds the carve via `fold_carve` and renders the substituted scope, so a carve's default is the default of its resulting scope), `type_to_string` (folded type for error messages), and `op_symbol`/`print_*_interval`. Per-domain rendering still lives in each domain file (`integer_to_string`, `bool_to_string`, …); this file composes them.
- **analyze.odin** (~1025 lines, formerly `analyzer.odin`) — Semantic analysis **logic** (the IR types it builds live in `ir.odin`). Builds a tree of `Scope_Type` (parallel `[dynamic]` arrays: `names`, `types` (constraints), `kind` (`Binding_Kind`), `values`, plus `type_folds`/`constraint_folds`). `walk()` is the AST→`Type` recursion, but is now a thin **dispatcher**: it switches on `Node_Kind` and routes each case to its own `walk_<kind>` handler (`walk_scope_node`, `walk_binding`, `walk_carve`, `walk_property`, `walk_operator`, `walk_literal`, …), one per branch — same split style as the Pratt parser's prefix/infix handlers. The handlers carry the per-kind logic; add a node kind by adding a `case` + a `walk_<kind>` proc. (`walk_operator` routes `&`/`|`/`~` to the set-algebra variants, `::` to a `Cast_Type` folded by `fold_cast`, and everything else to a `Compose_Type` folded by `fold_compose`.) `analyze()` is the entry; `follow()` chases references; `sem_error()` records an `Analyzer_Error` (the `Analyzer_Error`/`Analyzer_Error_Type` diagnostics stay here, with the analyzer state). The result lands in `cache.scope` / `cache.analyze_errors`. Two carve-related pieces live here: (1) `walk_property` resolves a **source-none property** (`.x`, `.#0`, `.x#0` — a self-mention written with no left side) against the scope being carved, reading its *original* field value, via `self_resolve` (a parent-less local lookup) and the `carved_scope` field on `Analyzer` (set/restored around each override walk in `walk_carve`, so nested carves resolve `.` against the nearest one); (2) `recheck_carve` folds a carve via `fold_carve` and re-proves every colored binding against its substituted value, raising `Constraint_Mismatch` for **implicit constraints** — a constraint on one field broken because another field it references was carved out of range (`u8:y -> x` then `a{x -> -11}`). The directly-overridden fields are still checked inline in `walk_carve`; `recheck_carve` covers the dependent ones.
- **integer.odin** (~800 lines, formerly `fold_integer.odin`) — Interval arithmetic over `Integer_Interval{lo, hi: Maybe(i64)}` (nil = ±∞), implementing `constraints.md`. Folds compose/range types to `[]Integer_Interval`, does interval add/sub/mul, union/intersect/negate/normalize, and `integer_intervals_satisfy` for the subset check (the constraint-as-contract proof). `builtin_name`/`builtin_alias` map intervals back to `u8`, `i32`, etc.; `int_layout` maps an interval to its `Int_Layout{bits, signed}` (the fixed-width builtins only). Also home to the **raw-cast bit layer** shared by `::` (see type.odin): `Bit_Repr{bits, width, signed, from_float}` is the little-endian pattern of any concrete value; `resize_bits` pads/cuts it (zero/sign-extend, truncate) to a target width; `bits_reinterpret_int` reads a resized pattern back as a two's-complement integer.
- **float.odin** (~629 lines) — The float domain, mirror of `integer.odin` over `[]Float_Interval`. Interval arithmetic and subset checks for the float family.
- **string.odin** (~1170 lines) — The string/char family over `[]String_Interval`. **Mode** is the interval's own `ordinal: bool` (set once at construction by `quotation_is_ordinal` from the literal's quote+length — single-quote ≤ 1 codepoint → ordinal `'a'..'z'`, everything else → positional `"a".."z"` prefix/suffix); `string_interval_mode` just reads the flag. There is no quotation stored in the interval anymore, and `String_Type` has **no `is_sequence` flag**. A `[]String_Interval` is **always a union** of alternatives (`|`). The ordered concatenation `+` is NOT flattened into a union: it stays a `Compose_Type` and is matched in order at satisfy time. `fold_string_sequence` collects the `+` chain's segments left-to-right (`"id_" + '0'..'9'*1..` → `[{"id_"}, {digits *1..}]`); `string_compose_satisfy`/`string_value_satisfies_sequence`/`seq_match` walk a concrete value, matching each segment its `count` times with backtracking. A **purely positional** `+` (every segment a concrete/prefix/suffix, no ordinal class, no repetition) still flattens to the legacy `{prefix, suffix}` interval via `positional_concat`/`concat_positional_fold`, so `~("".. + '_')` keeps working and `'a'..'z'+'@'+'a'..'z'` has a materializable default (`string_sequence_default`). **Negation in a sequence** works: an ordinal negation (`~'a'..'z'`) folds to codepoint-complement intervals (`string_intervals_negate`, consumable as a normal segment); a positional word negation (`~"piro"`, `~"x"`) becomes a SEGMENT-NEGATION sentinel — a `String_Interval` whose `count` is the impossible `{-1..-1}` (tagged by `seg_is_negation`/`count_negation_sentinel`) and whose `lo` is the forbidden word — which `seq_match` matches as a variable-length prefix that is NOT that word. The **three-bound range** `"ab".."cd".."ef"` stays a raw `Range_Type` (`string_is_tri_range`); `string_tri_range_satisfy` enforces all three: starts with the prefix, CONTAINS the middle, ends with the suffix. The compose/tri-range constraints are kept un-flattened by `fold_constraint` returning the raw node, and `satisfy` (type.odin) routes a `Compose_Type`/`Range_Type` `fc` to the sequence/tri matchers. `range_operand_kind` recognizes `Negate_Type`/`And_Type`/`Or_Type` bounds so `(~'\0')..` reads as a string range.
- **bool.odin** — The boolean domain, a finite set over `{true, false}`. `Bool_Type{value: Maybe(bool), default: bool}`: `value` is the concrete element for a singleton, `nil` for the full `{true,false}`; `default` is the materialized default — the **first source term** of the domain (`true` → true, `false` → false, `(true|false)` → true, `(false|true)` → false, the `bool` builtin → false). `&`/`|`/`~` are **set operations** (intersection/union/complement within `{true,false}`), not runtime logic: `true & false` → `none` (empty set), `(true|false)` → `{true,false}`, `~true` → `{false}`. The empty set folds to `None_Type`. Mirrors the string-family entry points (`fold_type_bool`/`fold_constraint_bool`/`bool_satisfy`).
- **type.odin** (~770 lines, formerly `fold_type.odin`) — Generic, domain-agnostic fold dispatch **only** (the `Type` rendering moved to `ir.odin`): `fold_type` derives the constraint a value *produces* (its envelope), routing to the integer/float/string/bool domains; `fold_constraint`/`fold_value_type`/`satisfy`/`type_default`/`default_value` resolve and check constraints. A collapse (`target!`, an `Execute_Type`) folds on both sides to the corresponding fold of the FIRST production it reduces through — `execute_production` resolves the target to a scope (peeling a carve) and returns its first `.Product` value, mirroring `reduce`/`execute`; `fold_constraint` and `fold_value_type` then recurse on it. Higher-level `fold_compose`/`fold_range` (validates range operand kinds and color matching — integer/float/string, no cross-family) plus type-shape checks. Emits `Invalid_operator`/`Invalid_Range` errors when operands can't fold. `fold_cast` resolves a `value :: target` **raw binary reinterpret-cast** (domain-agnostic): the target must fold to a single fixed-width layout (u8/i32/f64/bool/char or a string) via `cast_target`, else `Invalid_Cast`. `cast_to_bits` extracts the concrete source's little-endian bits (reading its width from its value, or from its color via `source_color`/`colored_int_layout`/`colored_float_kind` for a bare literal under a sized constraint); `cast_from_bits` resizes (pad/cut) and lays the pattern back down in the target domain. The result's envelope is always the target, so `fold_constraint`/`fold_value_type` return the target for a `Cast_Type` (the cast forces the value in — it never raises `Constraint_Mismatch`). Bit-for-bit for int/bool/string and float↔non-float; float→float of a different width is a **value** conversion (IEEE rounding, `f64 1.0 :: f32` → 1.0f) via `Bit_Repr.from_float`. So `(a+b)::u8` makes a u8-overflowing sum satisfy `u8:c`, and `??::u8` is an unknown whose envelope is `u8`. `fold_carve` materializes a `Carve_Type` into its substituted `Scope_Type` (clone the source scope, write each override into the field it targets + refresh that field's cached `type_fold`, then `repoint` every `Mention`/`Reference` that named the source scope so it names the copy — sibling references then read the substituted values, cascading transitively); it is pure (no diagnostics) and is wired into both `fold_type`/`fold_value_type` and `fold_constraint`. `repoint` is a copy-on-write rewrite of a `^Type` tree (descends through composites and nested scopes, clones a node only when a descendant changed). `scope_clone` copies a scope's columns; `new_type` boxes a `Type`.
- **pattern.odin** (~200 lines) — The pattern domain — `target ? { match -> product, … }` (the `Pattern_Type`/`Pattern_Branch` IR lives in ir.odin). A branch has two modes: **typecheck** `M -> p` (the match `M` is a constraint; fires when `type_fold(target) ⊆ M`) and **value** `=v -> p` (the match is a value; fires when `type_fold(target) ⊆ {-> v}` — `walk_pattern` peels the unary `=` operator and sets `value_match`). A `nil` match (`-> p`) is the default and matches anything. **Exhaustiveness** (`pattern_is_exhaustive`): a default branch, OR the OR of all branch matches typechecks against the target — `satisfy_root(fold_constraint(Or of covers), fold_value_type(target))`, where a value branch enters the union bundled as `{-> v}` (via `make_producer_scope`, `branch_match_cover`) and a typecheck branch enters as `M`. So `(0..120 | 22..255)` covers a u8 target just as the binding `(0..120|22..255):data -> ??::u8` typechecks; an uncovered target raises `Non_Exhaustive_Pattern`. **Folds**: `fold_constraint_pattern` → the product of the FIRST branch that covers the target (deterministic); `fold_type_pattern` → that same product when one branch covers (we know which fires at compile time), else the **combined** `Or` of every branch's product type (we don't, so the value is `a | b`). `branch_covers` is the per-branch firing test (`satisfy_root` of the branch's match against the target's type_fold). A Pattern_Type NEVER survives in a fold — it always resolves to a branch product (or its combined Or). `reduce_pattern` (reduce.odin) collapses to the firing branch's product; an unreachable fallback returns `Invalid_Type`, never the pattern.
- **diagnostics.odin** (~351 lines) — Turns analysis failures into author-facing messages. The fold procedures answer a yes/no question (does this resolve?); this layer explains *why* it didn't.
- **reduce.odin** (~440 lines) — Runtime reduction. `reduce(scope: ^Scope_Type) -> ^Type` collapses a scope through its `.Product` binding; `reduce_value` evaluates `Execute_Type` (collapse), `Compose_Type` (arithmetic), `Carve_Type`, references, etc. Prefers precomputed concrete `type_folds` when available.
- **resolve.odin** (~840 lines, formerly `resolver.odin`) — Compilation orchestrator. Thread pool for parallel files, `Cache` struct per file (`scope`, parse/analyze errors+warnings, `status`). Entry point `resolve_entry()`; the per-file pipeline that wires parse → analyze → (`--ir` print) → reduce → `print_type` is around line 370.
- **main.odin** — CLI; `parse_args()` into the `Options` struct. On-disk caching is force-disabled in the bootstrap (`no_cache = true`).
- **generate.odin**, **backends/x64|wasm|arm64/** — x64/codegen, WIP and not in the active pipeline.

Key conventions:
- **Builtin names are lowercase.** The generic-domain builtins are `int`, `float`, `string`, `bool`, `none` (registered in `init_builtins`), alongside the fixed-width `u8`/`i32`/`f64`/… The open-domain printers (`integer.odin`/`float.odin`/`string.odin`/`bool.odin`) print these same lowercase names, so source, IR, and diagnostics agree. There is no `Int`/`Bool`/`String` — a capitalized name like `Array` or `F32ORString` is always a user-defined scope, never a builtin.
- The constraint system is the heart of the analyzer. A concrete value `5` is the degenerate range `5..5`; `u8` is `0..255`; `true` is the singleton `{true}`. Constraints are **static contracts** — the analyzer must *prove* `value ⊆ constraint` (via `integer_intervals_satisfy`, `bool_satisfy`, …) or it's a `Constraint_Mismatch`. There is no implicit coercion; `&` is the explicit **narrowing** cast/intersection (it must still prove `value ⊆ result`). Distinct from it, `::` is the **raw binary reinterpret-cast**: `value :: target` forces the value's bits into the target's layout (pad/cut + reinterpret), so it always lands in the target and never needs a proof — `&` narrows, `::` reinterprets. Read `constraints.md` before changing folding.
- Same-name bindings are valid and tracked by ordinal (`#0`, `#1`, …) — a core feature, not a bug. Access (`.`) resolves the **last** occurrence; carving (`{}`) targets the **first** by default.
- Errors use the `Analyzer_Error_Type` enum and the term **constraint** (`Constraint_Mismatch`, `sem_error`-style naming), never "type"/"typecheck" — Syntact has no types.
- A constraint must denote a **statically-known set**. `Insoluble_Constraint` is raised when a constraint depends on an unknown value (`??`) — directly, through a reference, or inside any composition (`&`/`|`/`~`/arithmetic/range/carve/execute/pattern). The distinction is sharp: `u8` is the *set* `0..255` (solvable), but `??::u8` is *one indeterminate element* of u8, so `??::u8:a -> 10` cannot be proven. `constraint_depends_on_unknown` (analyze.odin) walks the raw constraint tree (following references, with a depth guard) and treats a `??` as solvable only when a `::` pins it to a concrete singleton. `Insoluble_Constraint` is checked in `typecheck` **before the `fc == nil` bail and the satisfaction proof**, so an unknown buried under any operator is caught even when the constraint folds to nil (e.g. a pattern whose target is a bare `??`) — but a `??` on the **value** side (`u8:a -> ??::u8`) is fine, only the constraint side must be solvable.
- **Or on the value side.** `satisfy(fc, ft)` decomposes a union on the VALUE side conjunctively: `a | b` satisfies `fc` iff BOTH `a` and `b` do (every value the union can take must land in `fc`). This mirrors the disjunctive Or on the constraint side and is what lets a pattern's combined product type `"" | 10` prove against the color `"" | 10`. Checked first, before the switch on `fc`.
- **Producer self-match vs raw productions.** `satisfy_root` folds each production it compares (`fold_constraint(v.values[i])`) before matching, because a producer scope built from a literal (`{->0..}`) stores its production RAW (a `Range_Type`), which `satisfy` cannot match against an `Integer`. Folding normalizes it to its domain set. This does NOT weaken the self-match rule: `u8:a -> u8` / `0..255:a -> 0..255` still fail (a set is not an element of itself), but `{->0..}:a -> 0..` (a producer of a set, satisfied by that set) now passes — needed for value-match exhaustiveness `0.. ? { =0.. }`.
- **Carve property values.** `reference_effective_value` (type.odin) resolves a `Reference_Type` honoring a carve in its target: a property `C.x` where `C` is `Base{x -> 10}` reads the OVERRIDE (10), not the original `Base.x` (5). `fold_value_type`'s Reference case uses it, so a carved field's value folds to the substituted value (mirrors `reduce_value`). `fold_constraint`'s Reference case keeps reading the original COLOR — a carve overrides the value, not the constraint.
- **Un-flattened string constraints.** A string `+` sequence and a three-bound string range do NOT fold to a flat `String_Type` — `fold_constraint` returns the raw `Compose_Type`/`Range_Type`, and `satisfy` has a `Compose_Type` case (→ `string_compose_satisfy`, ordered sequence) and a `Range_Type` case (→ `string_tri_range_satisfy`, prefix/contains/suffix). `fold_compose` does not diagnose a non-collapsing string `+` (it is a valid sequence). See string.odin.

### test/ package — five independent harnesses

`test/parse/`, `test/analyze/`, `test/typecheck/`, `test/reduce/`, `test/default/` are separate Odin test packages (packages `parse_test`/`analyze_test`/`typecheck_test`/`reduce_test`/`default_test`), each with `tests/*.json`, a `generated_tests.odin`, a `generator.odin`, and a runner. Run each with `odin test test/<category>` from the repo root (or `cd test/<category> && odin test .`) — the runner's `test_path` helper resolves `tests/*.json` against its own source directory, so the CWD no longer matters. The `generator.odin` regeneration step, by contrast, still reads/writes relative to the CWD and must run from inside the category directory.

- **parse** — JSON has `expect`, a serialized AST like `Scope[Operator(Add,Literal(Integer,1),Literal(Integer,2))]`. *Builds and runs* (1239 cases; ~10 fail on error-recovery / ambiguous-pattern cases).
- **analyze** — JSON has `expect_errors: []string` of `Analyzer_Error_Type` names (empty array = must analyze cleanly). The runner parses, analyzes, and compares the error list. *Builds and runs.*
- **typecheck** — same JSON shape as analyze (`expect_errors`), focused on constraint-satisfaction cases (`(string|u8):b -> "hi"`, `~10:a -> 5`, …), including carve **implicit-constraint** cases (`u8:y -> x` then `a{x -> -11}` → `Constraint_Mismatch`, transitive and `x+y` variants), **raw-cast** cases (`(a+b)::u8` satisfies `u8:c`; `??::u8`; `Invalid_Cast` for a layout-less target like `10..37`/`int`/`float`/disjoint union/negation), **composite** cases (mixed-family unions `(u8|f32|string)`/`(bool|u8)`, composed/double/triple negation `~(5|10)`/`~~~10`/`~(u8 & ~5)`/De Morgan `~(~(0..10) | 20)`, nested `&`/`|` of ranges and comparisons, **scope-shape** constraints `{u8:x}:a -> {x -> 5}` via `scope_satisfy`, computed scope fields `{u8:x -> 200, u8:y -> x+x}`, properties inside computations `pack.x + 50`), **self-match** cases (a non-singleton set never satisfies itself — `u8:a -> u8`, `string:a -> string`, `C -> u8 ; C:a -> C` all fail; a singleton does — `5:a -> 5`, `C -> 5 ; C:a -> C` pass), **carve** cases (carve as a constraint `Base{x->5}:a -> {x -> 5}`, via a reference `Narrowed -> Base{x->5}`, nested carves, reference-valued overrides `Base{x->k}`, dependent-field implicit checks, property of a carved scope), **execute** cases (`Base{x->5}!` as value and as constraint, collapse via reference chains `Q -> P ; Q!`, a producer of a SET vs a producer of a concrete element), and **references composed in `&`/`|`/`~` and arithmetic** (`(C & >10)`, `(C | string)`, `~C`, `A + B` under a constraint), plus the `tc_pat_*` **pattern** cases and the `tc_str_neg_*`/`tc_str_tri_range_*` **string sequence/negation/three-bound-range** cases. *Builds and runs — all 460 pass.* (The string gaps that `constraints.md` flagged are now implemented, not just exposed.)
- **reduce** — JSON has `expect`, the stringified reduced value (e.g. `"25"`). The runner calls `reduce(cache.scope)` then `value_to_string`. *Builds and runs*; some cases still fail on unimplemented reducer features (see "Current state").
- **default** — JSON has `binding` (a name) + `expect` (the materialized default, rendered by `value_to_string`). The runner analyzes, finds the **last** binding of that name in the root scope, and compares its `type_folds` value (or `values` if unfolded) to `expect`. Covers the default-materialization rule (range → low bound, union `|` → first term's default, negation `~X` → first element outside X, scope → `{name -> default, …}`, carve → default of the substituted scope, reference-as-type → default of the referenced set, execute → default of the produced value) across all domains and composites, including carve-via-reference, nested carves, and collapse-via-reference. *Builds and runs — all 109 pass.* The materialization gaps it used to expose are now fixed (cross-family union defaults via `type_default`'s `Or_Type` first-term rule, computed scope/carve fields via `write_value` preferring `type_folds` and `fold_carve` re-folding dependents, empty-set negation → `none`, string repetition/quoting, and string negation → first string outside the set).

Test JSON shape:
```json
{ "name": "...", "description": "...", "source": "u8:a -> 200\n-> a", "expect_errors": [] }
```
Test function names are `test_{stem}_{index}`; find the exact name in the category's `generated_tests.odin`.
